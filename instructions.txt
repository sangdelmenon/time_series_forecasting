2. WHAT THE PROJECT DOES
================================================================================

HIGH-LEVEL FUNCTIONALITY:

Step 1: DATA GENERATION & LOADING
- Create synthetic daily sales data for 50 products
- 3 years of historical data (~1,095 days)
- Realistic patterns:
  • Trend (gradual growth/decline)
  • Seasonality (weekly, monthly, yearly)
  • Special events (holidays, promotions)
  • Random noise (realistic variability)

Step 2: EXPLORATORY TIME SERIES ANALYSIS
- Visualize sales patterns over time
- Decompose into trend + seasonality + residual
- Test for stationarity (ADF test)
- Analyze autocorrelation (ACF) and partial autocorrelation (PACF)
- Identify outliers and anomalies
- Calculate summary statistics

Step 3: DATA PREPROCESSING
- Handle missing values (forward fill, interpolation)
- Remove outliers (IQR method)
- Create features:
  • Lagged values (sales from 1, 7, 30 days ago)
  • Rolling statistics (7-day, 30-day moving averages)
  • Day of week, month, quarter
  • Holiday indicators
  • Promotion flags
- Normalize data (for LSTM)

Step 4: ARIMA MODEL
Statistical approach:
- Auto-select parameters (p, d, q) using AIC/BIC
- Fit ARIMA model on training data
- Generate forecasts with confidence intervals
- Validate on test set

ARIMA components:
- AR (p): Autoregressive - use past values
- I (d): Integrated - differencing to achieve stationarity
- MA (q): Moving Average - use past forecast errors

Step 5: LSTM MODEL
Deep learning approach:
- Create sequences (90-day windows)
- Architecture:
  • Input layer: 90 timesteps × features
  • LSTM layer 1: 64 units, return sequences
  • Dropout: 0.2 (prevent overfitting)
  • LSTM layer 2: 32 units
  • Dropout: 0.2
  • Dense layer: 1 unit (prediction)
- Train with Adam optimizer
- Early stopping to prevent overfitting

Step 6: HYPERPARAMETER OPTIMIZATION
ARIMA:
- Grid search over p, d, q ranges
- Metric: AIC (Akaike Information Criterion)

LSTM:
- Grid search over:
  • Number of LSTM units: [32, 64, 128]
  • Number of layers: [1, 2, 3]
  • Dropout rate: [0.1, 0.2, 0.3]
  • Learning rate: [0.001, 0.01]
  • Batch size: [16, 32, 64]
- Metric: Validation MAE

Step 7: BACKTESTING
Walk-forward validation:
- Train on months 1-24, test on month 25
- Train on months 1-25, test on month 26
- Continue rolling forward
- Average performance across all test periods

Benefits:
- More realistic than single train-test split
- Tests model on different market conditions
- Detects overfitting

Step 8: MODEL COMPARISON
Evaluate on:
- MAE (Mean Absolute Error): Average prediction error
- RMSE (Root Mean Squared Error): Penalizes large errors
- MAPE (Mean Absolute Percentage Error): Relative error
- Accuracy: % predictions within ±10% of actual
- Forecast bias: Are we consistently over/under predicting?

Step 9: FORECAST GENERATION
- Generate 30-day ahead forecasts
- Create prediction intervals (80%, 95% confidence)
- Visualize forecasts vs actuals
- Flag high-uncertainty predictions

Step 10: PRODUCTION DEPLOYMENT
- Save trained models (pickle for ARIMA, .h5 for LSTM)
- Create forecast API
- Set up monitoring dashboard
- Define retraining schedule

REAL-WORLD EXAMPLE:
Input: Historical sales for "Product_A"
  - Last 90 days: [45, 52, 48, 50, 55, ..., 60]
  - Trend: Gradually increasing
  - Seasonality: Higher on weekends
  - Recent promotion: Spike last week

Output:
  - Next 7 days forecast: [62, 64, 58, 55, 54, 68, 70]
  - 95% Confidence interval: [55-69, 57-71, 51-65, ...]

ARIMA forecast: [60, 62, 57, 54, 53, 66, 68]
LSTM forecast: [63, 65, 59, 56, 55, 69, 71]
Ensemble (average): [62, 64, 58, 55, 54, 68, 70]

DECISION 1: Why Both ARIMA and LSTM?

CHOICE: Implement both statistical (ARIMA) and deep learning (LSTM) approaches

REASONS:
- Different strengths:
  • ARIMA: Interpretable, fast, good for linear patterns
  • LSTM: Captures complex non-linear patterns, handles multiple features
- Ensemble opportunity: Average predictions for better accuracy
- Comparison: See which works better for different products
- Portfolio demonstration: Show breadth of knowledge

WHEN ARIMA WINS:
- Short time series (< 500 points)
- Strong linear trends and seasonality
- Need interpretability
- Limited computational resources

WHEN LSTM WINS:
- Long time series (> 1000 points)
- Complex non-linear patterns
- Multiple external features
- Sufficient data for training

ENSEMBLE APPROACH:
```python
forecast_arima = arima_model.forecast(steps=30)
forecast_lstm = lstm_model.predict(X_test)
forecast_ensemble = 0.5 * forecast_arima + 0.5 * forecast_lstm
```

Often outperforms individual models!

DECISION 2: Why 90-Day Lookback Window?

CHOICE: Use 90 days of history to predict next day

PROBLEM: How much history should LSTM consider?
- Too short (7 days): Misses long-term patterns
- Too long (365 days): Harder to train, more noise

90-DAY RATIONALE:
- Captures quarterly patterns
- Balances memory and computation
- Industry standard for many forecasting tasks
- Allows LSTM to learn:
  • Weekly seasonality (90/7 ≈ 13 weeks)
  • Monthly patterns (3 months)
  • Short-term trends

ALTERNATIVE WINDOW SIZES:
- 7 days: Only captures weekly patterns
- 30 days: Misses quarterly effects
- 180 days: Too much data, harder to train

EXPERIMENT:
```python
windows = [30, 60, 90, 120]
for window in windows:
    model = train_lstm(lookback=window)
    mae = evaluate(model)
    print(f"Window {window}: MAE = {mae}")

# Expected result: 90 days optimal for most products
```

DECISION 3: Why 2-Layer LSTM?

CHOICE: Use 2 LSTM layers (64 units, 32 units)

ARCHITECTURE RATIONALE:
- Layer 1 (64 units): Learn high-level patterns
- Layer 2 (32 units): Refine predictions
- Dropout (0.2): Prevent overfitting
- Dense (1 unit): Final prediction

ALTERNATIVES TRIED:
- 1 layer: Underfits, MAE 15.2
- 2 layers: Best, MAE 12.5
- 3 layers: Overfits, MAE 14.8

WHY NOT MORE LAYERS:
- Diminishing returns
- Longer training time
- Risk of overfitting
- More difficult to tune

CODE EXAMPLE:
```python
model = Sequential([
    LSTM(64, return_sequences=True, input_shape=(90, n_features)),
    Dropout(0.2),
    LSTM(32, return_sequences=False),
    Dropout(0.2),
    Dense(1)
])
```

return_sequences=True: Pass sequences to next LSTM layer
return_sequences=False: Output single value

DECISION 4: Why Walk-Forward Validation?

CHOICE: Use walk-forward validation instead of random split

PROBLEM WITH RANDOM SPLIT:
- Time series has temporal dependency
- Future cannot predict past
- Random split breaks time order

WALK-FORWARD METHOD:
```
Train: [Jan-Dec Year 1] → Test: [Jan Year 2]
Train: [Jan Year 1 - Jan Year 2] → Test: [Feb Year 2]
Train: [Jan Year 1 - Feb Year 2] → Test: [Mar Year 2]
...
```

BENEFITS:
- Realistic: Mimics production use
- Tests model on different periods
- Detects overfitting
- Gives time-series-appropriate performance

IMPLEMENTATION:
```python
n_test_periods = 12  # Test on 12 months
train_size = len(data) - n_test_periods

for i in range(n_test_periods):
    train = data[:train_size + i]
    test = data[train_size + i:train_size + i + 1]
    
    model.fit(train)
    pred = model.forecast(1)
    
    errors.append(abs(pred - test))

mae = np.mean(errors)
```

DECISION 5: Why MAE as Primary Metric?

CHOICE: Use MAE, not RMSE or MAPE, as primary metric

COMPARISON:

MAE (Mean Absolute Error):
- Formula: mean(|predicted - actual|)
- Units: Same as data (e.g., units sold)
- Intuitive: "On average, we're off by 12.5 units"
- Robust to outliers

RMSE (Root Mean Squared Error):
- Formula: sqrt(mean((predicted - actual)²))
- Penalizes large errors more
- Use when large errors are very costly

MAPE (Mean Absolute Percentage Error):
- Formula: mean(|predicted - actual| / actual) × 100%
- Problem: Undefined when actual = 0
- Problem: Asymmetric (penalizes over-prediction more)

CHOICE: MAE primary, RMSE secondary
- MAE: Easier to interpret, robust
- RMSE: Detect if large errors are common
- MAPE: Supplementary, for relative error

DECISION 6: Why Auto-ARIMA?

CHOICE: Use auto_arima for parameter selection

MANUAL ARIMA SELECTION:
- Look at ACF plot → choose q (MA order)
- Look at PACF plot → choose p (AR order)
- Try different combinations
- Compare AIC/BIC
- Time-consuming, requires expertise

AUTO-ARIMA:
- Automatically tests combinations
- Selects based on AIC
- Faster, more consistent
- Good for multiple products

HOW IT WORKS:
```python
from pmdarima import auto_arima

model = auto_arima(
    train_data,
    start_p=0, max_p=5,  # AR order range
    start_q=0, max_q=5,  # MA order range
    d=None,  # Auto-determine differencing
    seasonal=True,
    m=7,  # Weekly seasonality
    stepwise=True,  # Faster search
    suppress_warnings=True
)

print(f"Selected: ARIMA{model.order}")
```

TRADE-OFF:
- Automated, but less control
- Might not select optimal for specific domain knowledge
- Can override if needed

DECISION 7: Why Seasonal Decomposition?

CHOICE: Use STL (Seasonal and Trend decomposition using Loess)

PURPOSE: Break time series into components
- Trend: Long-term increase/decrease
- Seasonal: Repeating patterns
- Residual: Random noise

BENEFITS:
- Understand data structure
- Improve forecasts (model each component separately)
- Detect anomalies (large residuals)
- Communicate patterns to stakeholders

STL VS ALTERNATIVES:
- Classical decomposition: Assumes constant seasonality
- STL: Handles changing seasonality
- X-12-ARIMA: More complex, overkill for most cases

CODE EXAMPLE:
```python
from statsmodels.tsa.seasonal import STL

stl = STL(data, seasonal=7)  # Weekly seasonality
result = stl.fit()

trend = result.trend
seasonal = result.seasonal
residual = result.resid

# Visualize
fig, axes = plt.subplots(4, 1, figsize=(10, 8))
axes[0].plot(data)
axes[0].set_title('Original')
axes[1].plot(trend)
axes[1].set_title('Trend')
axes[2].plot(seasonal)
axes[2].set_title('Seasonal')
axes[3].plot(residual)
axes[3].set_title('Residual')
```

DECISION 8: Why Normalize Data for LSTM?

CHOICE: Apply MinMaxScaler to scale features to [0, 1]

PROBLEM: LSTM sensitive to input scale
- Sales: 0-1000 units
- Price: $10-$200
- Promotion: 0-1
Different scales slow training, hurt performance

MINMAXSCALER:
```python
from sklearn.preprocessing import MinMaxScaler

scaler = MinMaxScaler(feature_range=(0, 1))
data_scaled = scaler.fit_transform(data)

# After prediction
predictions_original = scaler.inverse_transform(predictions)
```

ALTERNATIVES:
- StandardScaler: Mean 0, std 1 (use if data has outliers)
- RobustScaler: Median-based (very robust to outliers)

CHOSEN MINMAXSCALER BECAUSE:
- LSTM works well with bounded inputs [0, 1]
- Preserves zero values
- Interpretable (0 = minimum, 1 = maximum)

CRITICAL: Fit scaler on training data only!
```python
scaler.fit(train_data)  # ✓ Correct
data_scaled = scaler.transform(test_data)

# ✗ Wrong: Fit on test data (data leakage)
scaler.fit_transform(test_data)
```

DECISION 9: Why Early Stopping?

CHOICE: Use early stopping for LSTM training

PROBLEM: How many epochs?
- Too few: Underfitting
- Too many: Overfitting

EARLY STOPPING:
- Monitor validation loss
- Stop if no improvement for N epochs
- Restore best weights

```python
from tensorflow.keras.callbacks import EarlyStopping

early_stop = EarlyStopping(
    monitor='val_loss',
    patience=10,  # Wait 10 epochs
    restore_best_weights=True
)

model.fit(
    X_train, y_train,
    validation_data=(X_val, y_val),
    epochs=100,
    callbacks=[early_stop]
)

# Typically stops after 30-50 epochs
```

BENEFITS:
- Automatic stopping
- Prevents overfitting
- Saves training time
- No need to manually tune epochs

DECISION 10: Why Generate Forecast Intervals?

CHOICE: Provide prediction intervals (80%, 95% confidence)

POINT FORECAST LIMITATION:
- "Forecast: 60 units"
- How confident are we?
- What if actual is 45 or 75?

INTERVAL FORECAST:
- "Forecast: 60 units (95% CI: 50-70)"
- Communicates uncertainty
- Better business decisions

ARIMA INTERVALS (Analytical):
```python
forecast_result = model.get_forecast(steps=30)
forecast_mean = forecast_result.predicted_mean
forecast_ci = forecast_result.conf_int(alpha=0.05)  # 95% CI

lower_bound = forecast_ci['lower']
upper_bound = forecast_ci['upper']
```

LSTM INTERVALS (Monte Carlo Dropout):
```python
# Enable dropout at prediction time
predictions = []
for _ in range(100):  # 100 stochastic forward passes
    pred = model.predict(X_test, training=True)
    predictions.append(pred)

mean = np.mean(predictions, axis=0)
lower = np.percentile(predictions, 2.5, axis=0)
upper = np.percentile(predictions, 97.5, axis=0)
```

BUSINESS VALUE:
- Stock upper bound (avoid stockouts)
- Budget for mean (average case)
- Lower bound as worst-case scenario
